= Premigration checklists
include::_attributes/common-attributes.adoc[]
:context: ossm-migrating-premigration-checklists-assembly

toc::[]

These checklists make the following assumptions:

1. You have read "Important information to know before migrating" and understand the differences between {SMProduct} 2 and {SMProduct} 3.
2. You have read through the migration recommendations.
//migrating/ossm-migrating-from-2-to-3-assembly.html#recommendations_ossm-migrating-from-2-to-3-assembly
3. You are familiar with the `ServiceMeshControlPlane` resources no longer found in the `istio` resource.
4. You want to migrate from {SMProduct} 2 to {SMProduct} 3.
5. You are running {SMProduct} {SMv2Version}.
6. You have upgraded your `ServiceMeshControlPlane` resource to the latest version.
7. If you are using {KialiProduct}, you are running the latest version.
8. You have installed the {SMProduct} Operator 3.

To install the {SMProduct} 3 Operator, see: xref:../../install/ossm-installing-openshift-service-mesh.adoc[Installing OpenShift Service Mesh]

[NOTE]
====
You must complete each checklist before you can begin migrating.
====

== Checklist for configuring add on replacements in Service Mesh 2

After you have installed {SMProduct} Operator 3, if you have not already done so when you upgraded from {SMProduct} 2.5 to {SMProduct} 2.6, you need to disable and configure a replacement for the following add ons in your `ServiceMeshControlPlane` resource in {SMProduct} 2:

* [ ] Disable Prometheus: `spec.addons.prometheus.enabled=false`
** [ ] Configure the `ServiceMeshControlPlane` with OpenShift Monitoring as the replacement. These instructions also include installing a standalone `Kiali` resource. Both can be done at the same time. link:https://docs.redhat.com/en/documentation/openshift_container_platform/4.17/html/service_mesh/service-mesh-2-x#ossm-integrating-with-user-workload-monitoring_observability[Integration user-workload monitoring,window=_blank]
** [ ] If you are not using OpenShift monitoring, see: link:https://docs.redhat.com/en/documentation/red_hat_openshift_service_on_aws/4/html/service_mesh/service-mesh-2-x#integration-with-external-prometheus-installation[Integration with external Prometheus installation]
* [ ] Disable Tracing: `spec.tracing.type=None`
** [ ] Configure the `ServiceMeshControlPlane` with OpenShift Distributed Tracing as the replacement. link:https://docs.redhat.com/en/documentation/openshift_container_platform/4.17/html/service_mesh/service-mesh-2-x#ossm-configuring-distr-tracing-tempo_observability[Configuring Distributed Tracing Tempo]
* [ ] Disable Kiali: `spec.addons.kiali.enabled=false`
** [ ] Create a standalone `Kiali` resource as the replacement. If you haven't already done this as part of the previous items for Prometheus or Tracing, follow link:https://docs.redhat.com/en/documentation/red_hat_openshift_service_mesh/3.0.0tp1/html/observability/kiali-operator-provided-by-red-hat#ossm-install-kiali-operator_ossm-kiali-assembly[Kiali Operator provided by Red Hat] on how to install and configure Kiali as a standalone resource.

== Checklist for migrating {SMProduct} 2 routing and gateways

This section applies if you have not already moved to explicitly-managed routes and gateway injection.

Remember that automatic route creation, also known as Istio OpenShift Routing (IOR), is a deprecated feature that is disabled by default for any `ServiceMeshControlPlane` resource that was created using {SMProduct} 2.5 and later. To move from {SMProduct} 2 to {SMProduct} 3, you need to migrate from IOR to explicitly-managed routes.

* [ ] Migrate from Istio OpenShift Routing (IOR) to to explicitly-managed routes. link:https://docs.redhat.com/en/documentation/openshift_container_platform/4.17/html/service_mesh/service-mesh-2-x#ossm-route-migration[Service Mesh Route Migration]

Since the {SMProduct} 3 control plane does not manage gateways, you must manage them yourself. To do so, and move from {SMProduct} 2 to {SMProduct} 3, you need to disable the default gateways, move away from the Service Mesh Control Plane declaration, and begin to use gateway injection.

* [ ] Migrate to gateway injection link:https://docs.redhat.com/en/documentation/openshift_container_platform/4.17/html/service_mesh/service-mesh-2-x#ossm-gateway-migration[Service Mesh Gateway Migration] to migrate off of the default ingress/egress gateways.

== Checklist for disabling network policy management and Grafana in {SMProduct} 2

It is recommended to disable network policies management in the {SMProduct} 2 `ServiceMeshControlPlane` resource before you start your migration, and recreate the network policies after you have migrate your deployment and workloads.

However, if you cannot recreate your network policies after migrating due to security concerns, you must recreate your network policies first, and then set `spec.security.manageNetworkPolicy` to `false`. <insert exref>[Migrating network policies for security purposes]

  * [ ] Disable network policy management: `spec.security.manageNetworkPolicy=false`.

Grafana is not supported in {SMProduct} 3, and must be disabled in your {SMProduct} 2 `ServiceMeshControlPlane`:

  * [ ] Disable Grafana: `spec.addons.grafana.enabled=false`.

== Check your deployment model

Your deployment model determines your migration path. Run the following command to determine if you are running a `MultiTenant` or a `ClusterWide` mesh:

[source,terminal]
----
oc get smcp <smcp-name> -n <smcp-namespace> -o jsonpath='{.spec.mode}'
----

If you are not using cert-manager, then continue to the appropriate next step for your deployment model.
