= About Network Observability Operator FlowCollectorSlice

[role="_abstract"]
<insert content>

`FlowCollectorSlice` is a Network Observability Opertar custom resource (CR) that allows you to define logical slices of network flow collection in a cluster. Each slice represents a scoped subset of network traffic selected by namespace, subnet, or label-based criteria. It can optionally apply custom sampling behavior

This mechanism complements the existing `FlowCollector` custom resource by enabling granular, selective, and multi-tenant-aware flow collection, instead of a single global configuration that applies uniformly to all traffic.

When slice-based collection is enabled, only traffic that matches at least one `FlowCollectorSlice` is collected, allowing administrators to precisely control which network flows are observed.

== Benefits of FlowCollectorSlice

By default, network flow collection applies uniformly to all traffic in the cluster. This can result in excessive data volume and limited flexibility.

Using `FlowCollectorSlice` provides the following benefits:

* Enables selective flow collection for specific namespaces or workloads
* Supports multi-tenant and environment-based observability
* Reduces storage and processing costs by filtering irrelevant traffic
* Preserves backward compatibility through opt-in configuration

== Relationship between FlowCollector and FlowCollectorSlice

* `FlowCollector` defines *global flow collection behavior* for the cluster.
* `FlowCollectorSlice` defines *which traffic is eligible for collection* when slice-based filtering is enabled.
* The `FlowCollector.spec.slicesConfig` field controls how slice definitions are applied.

== Collection modes

Slice-based collection behavior is controlled by the `collectionMode` field.

[cols="1,3",options="header"]
|===
| Mode | Description

| `AlwaysCollect`
| Collects all traffic and ignores FlowCollectorSlice resources.

| `AllowList`
| Collects only traffic that matches at least one FlowCollectorSlice or an explicitly allowed namespace.
|===

== FlowCollectorSlice status

Each FlowCollectorSlice resource exposes a status `subresource` that reports:

* Validation results
* Reconciliation state
* Whether the slice is successfully applied

This status allows administrators to verify that slice definitions are active and functioning as expected.


=== Motivation

Traditional flow collection operates at cluster scope, which can lead to:

* Excessive data volume and storage costs
* Limited ability to target specific tenants, applications, or environments
* Difficulty enforcing differentiated observability policies

The FlowCollectorSlice API addresses these limitations by:

* Allowing *explicit opt-in* to flow collection for selected namespaces or traffic classes
* Supporting *multiple concurrent collection policies*
* Enabling *future extensions* such as per-slice sampling rates, labeling, or processing

=== Relationship Between FlowCollector and FlowCollectorSlice

* *FlowCollector* remains the _global controller_ for flow collection behavior.
* *FlowCollectorSlice* resources define _what traffic is included or excluded_ when slice-based collection is enabled.
* A new `slicesConfig` field in the FlowCollector custom resource controls:
** Whether slice-based collection is enabled
** How slices affect the overall collection mode

The operator reconciles FlowCollectorSlice resources and translates them into runtime configuration for flow collection components.

=== Collection Modes

Slice behavior is governed by `FlowCollector.spec.slicesConfig.collectionMode`:

* *AlwaysCollect*
** FlowCollectorSlice resources are ignored.
** All traffic continues to be collected.
** This mode preserves backward compatibility.

* *AllowList*
** Only traffic matching at least one FlowCollectorSlice is collected.
** An optional namespace allow list can be used to always include selected namespaces.

This design allows administrators to progressively adopt slice-based collection without disrupting existing deployments.

=== Status and Observability

Each FlowCollectorSlice exposes a status subresource that reports:

* Validation and reconciliation conditions
* Whether the slice has been successfully applied
* Any errors related to filter resolution

This ensures visibility into how slice definitions affect actual flow collection.
